### 一、Hermes 是什么

**Hermes** 是 **Nous Research** 开源的 **self-improving personal agent**

这个定义可以拆成三层：

| 层次 | 含义 |
|---|---|
| **Agent** | 不只是模型问答，而是带工具、状态、记忆、控制循环的运行时系统 |
| **Personal** | 面向单个用户的长期工作代理，不优先追求多租户和团队协作 |
| **Self-Improving** | 能从任务经验中沉淀技能和记忆，让后续任务受益 |

更工程化地看，**Hermes** 可以概括为：

```text
Hermes = LLM + Tools + Skills + Memory + Session + Gateway + Runtime Defense
```

它不是以下几类东西：

| 不是 | 原因 |
|---|---|
| 普通 **chatbot** | 普通聊天主要围绕单次上下文生成回复，**Hermes** 有工具、记忆、状态和持续运行能力 |
| 单纯 **tool calling demo** | 工具调用只是能力层的一部分，**Hermes** 还包含技能沉淀、会话持久化、平台接入和运行时防御 |
| 团队协作 **SaaS** | **Hermes** 的默认取舍是个人长期代理，而不是组织级权限、多租户和团队空间 |
| 只服务编码任务的 **coding agent** | 编码只是能力之一，**Hermes** 的目标是通用个人工作代理 |

当前阶段需要记住的核心定义：

```text
Hermes 是一个围绕个人长期协作设计的 Agent Runtime
它把 LLM 决策、工具执行、经验沉淀、长期记忆和多入口接入放在同一个运行时系统里
```

### 二、Hermes 的核心设计亮点

#### 2.1、从经验沉淀技能

**Hermes** 不只依赖开发者预先写好的 **prompt**、工具和规则

它会尝试把复杂任务中的可复用过程沉淀为 **skill**，让后续类似任务能复用已有经验

工程后果：

- 需要有技能文件的存储、检索、更新机制
- 需要有触发创建技能的时机判断
- 需要把技能索引注入上下文，但不能让上下文无限膨胀
- 需要处理低质量技能误导后续任务的风险

#### 2.2、终端优先

**CLI-First** 的含义不是“只有命令行”，而是终端体验被当成核心产品来设计

这会影响运行时的基本形态：

- 文件系统成为天然的一等存储位置
- 本地 **SQLite** 比远程数据库更贴合默认部署
- 工具执行和命令运行天然靠近终端
- 单进程模型比复杂微服务更符合低成本运行目标

#### 2.3、个人长期记忆

**Hermes** 的记忆不是普通聊天记录的简单保留，而是围绕用户长期工作关系进行组织

它要记住的不只是“说过什么”，还包括：

- 用户偏好
- 工作习惯
- 已沉淀的技能
- 长期任务上下文
- 跨会话的历史状态

工程后果：

- 需要区分短期会话状态和长期记忆
- 需要在上下文中选择性注入记忆
- 需要防止错误记忆、过期偏好和上下文污染
- 需要明确个人数据边界

#### 2.4、多环境运行

**Run Anywhere** 的重点不是口号，而是约束：系统不能默认依赖重型外部基础设施

这会推动 **Hermes** 选择：

- 轻量部署
- 可替换模型 **provider**
- 本地文件和 **SQLite**
- 多种终端执行后端
- **Gateway** 平台投射
- fallback、timeout、context compression、iteration budget 等防御机制

### 三、四个设计赌注

所谓设计赌注，是早期确定、后续很难逆转、会影响全局架构的选择

**Hermes** 的四个赌注不是宣传语，而是解释后续源码结构的主线

#### 3.1、Learning Loop

**定义**

**Learning Loop** 表示 **Agent** 能从执行任务的经验中提炼可复用知识，并在后续任务中主动使用这些知识

**架构含义**

普通 **Agent** 常见结构是：

```text
用户输入 -> 拼上下文 -> 调模型 -> 调工具 -> 返回结果
```

**Hermes** 多了一条经验回流路径：

```text
用户输入 -> 执行任务 -> 识别可复用经验 -> 写入 skill / memory -> 后续任务检索复用
```

**典型机制**

- **skills**：保存可复用过程知识
- **Memory Nudge**：周期性提醒是否需要更新记忆或创建技能
- **Background Review**：后台审查任务轨迹，尝试总结经验

**代价**

- skill 索引会增加上下文 token 成本
- 后台审查会带来额外模型调用成本
- 低质量技能可能污染后续执行
- 学习行为需要被约束，否则会把偶然经验固化为错误规则

**复习时要问的问题**

- 这段源码是否在处理经验沉淀
- 它沉淀的是 skill、memory，还是 session 状态
- 它如何避免把错误经验长期保存

#### 3.2、CLI-First

**定义**

**CLI-First** 表示终端不是备选入口，而是 **Hermes** 的核心交互和运行环境

**架构含义**

终端优先会让整个系统更接近本地运行时：

```text
命令行入口
  -> 本地配置
  -> 本地 session / memory
  -> 本地工具执行
  -> 可选 Gateway 投射
```

**典型机制**

- **CLI / TUI** 入口
- 文件系统和 **SQLite** 状态管理
- terminal tool
- 多种 terminal backend
- 本地 profile 和配置文件

**代价**

- 终端交互复杂度高，尤其是流式输出、宽度限制、键盘交互和进程控制
- **CLI** 代码容易变得很大
- 图片、表格、富文本体验弱于 **Web UI**
- 远程运行时需要额外平台投射方案

**复习时要问的问题**

- 这个设计是否服务于终端优先
- 它是否依赖本地文件系统或本地进程模型
- 如果换成纯 **Web SaaS**，这段逻辑是否会完全不同

#### 3.3、Personal Long-Term

**定义**

**Personal Long-Term** 表示 **Hermes** 面向单个用户的长期协作关系，而不是短期会话或多人平台

**架构含义**

系统默认围绕个人数据和长期上下文组织：

```text
用户偏好
  + 长期记忆
  + 会话历史
  + 个人技能
  + profile 配置
  -> 个性化 Agent 行为
```

**典型机制**

- **SessionDB**
- **Memory Provider**
- **USER.md / MEMORY.md**
- profile 隔离
- 个人 skill 文件

**代价**

- 多租户能力不是默认优势
- 团队共享、权限分层和组织治理需要额外设计
- 个人长期记忆会带来隐私和错误固化风险
- 记忆越丰富，越需要检索、压缩和边界控制

**复习时要问的问题**

- 这段逻辑保存的是短期状态还是长期记忆
- 它是否应该跨 session 生效
- 它是否可能泄露或污染个人上下文

#### 3.4、Run Anywhere

**定义**

**Run Anywhere** 表示 **Hermes** 要能在本地、远程服务器、低成本 **VPS**、容器、消息平台等环境中运行

**架构含义**

系统必须控制依赖、资源占用和环境假设：

```text
轻量运行时
  + 本地持久化
  + 可替换 provider
  + 多终端后端
  + Gateway
  + 防御机制
```

**典型机制**

- fallback provider
- context compression
- iteration budget
- terminal backend 抽象
- **Gateway** 平台适配
- 资源生命周期管理

**代价**

- 不同运行环境差异大，调试复杂
- **SQLite** 并发能力有限，需要应用层重试和规避
- 远程执行环境需要处理网络、认证、资源释放
- 多平台适配会带来大量边界代码

**复习时要问的问题**

- 这段逻辑是否假设只能本地运行
- 它如何处理资源不足、模型失败、工具失败
- 它是否能在远程或消息平台入口下继续工作

### 四、Self-Improving 的工程闭环

**Self-Improving** 不能理解成模型自己变强

在 **Hermes** 里，它更接近运行时层面的经验闭环：

```text
任务执行
  -> 产生工具调用轨迹和对话上下文
  -> 判断是否存在可复用经验
  -> 更新 skill / memory
  -> 后续任务检索相关经验
  -> 使用经验提升执行效率和稳定性
```

#### 4.1、Skill System

**Skill System** 负责把过程性知识保存成可复用文件

它记录的不是“某个事实”，而是“遇到某类任务时应该如何做”

典型用途：

- 自动化步骤
- 项目约定
- 工具组合方式
- 领域任务流程

风险：

- 把一次偶然任务经验保存为通用规则
- 技能过多导致检索和选择困难
- 技能描述不清晰导致模型误用

#### 4.2、Memory Nudge

**Memory Nudge** 是周期性提醒机制

它的作用不是直接写记忆，而是在合适时机提醒 **Agent** 思考是否需要更新记忆或创建技能

可以理解为：

```text
不是每一步都学习
而是在一定轮次或工具调用后检查是否值得学习
```

这个机制的意义是控制学习频率，避免每次交互都产生记忆噪音

#### 4.3、Background Review

**Background Review** 是后台审查机制

它把任务轨迹交给一个受限的后台审查过程，用来总结可沉淀内容

关键点：

- 它不阻塞主任务完成
- 它需要独立预算，防止无限消耗资源
- 它应该共享必要记忆，但不能破坏主任务控制流

#### 4.4、闭环判断

判断一个 **Agent** 是否真正 self-improving，不能只看它有没有 memory

更关键的问题是：

- 它是否知道什么时候应该学习
- 它学习的是事实、偏好，还是过程性技能
- 它是否能在后续任务中检索并使用
- 它是否能修正或淘汰低质量经验

### 五、Hermes 五层架构模型

第 2 章给出的五层模型，是后续读源码时最重要的定位工具

```text
入口层：用户如何触达 Agent
编排层：Agent 如何决策和循环
能力层：Agent 能调用什么
状态层：Agent 记住什么
平台层：Agent 如何投射到不同平台
```

#### 5.1、入口层

入口层解决“请求从哪里进入”

典型入口：

- **CLI**
- **TUI**
- **Gateway**
- **Cron**
- **Batch**

职责：

- 接收用户输入
- 解析命令和参数
- 加载 profile 和配置
- 创建或恢复 session
- 把请求交给编排层

不应该负责：

- 模型推理策略
- 工具选择逻辑
- 长期记忆决策
- 平台协议细节之外的 Agent 行为

读源码时的定位方式：

```text
问题和命令行、启动参数、交互入口有关 -> 先看入口层
```

#### 5.2、编排层

编排层是 **Hermes** 的心脏

它解决“**Agent** 如何组织一次任务执行”

核心职责：

- 初始化 **AIAgent**
- 构造 system prompt
- 注入 memory 和 skills
- 调用模型 provider
- 暴露和调度工具
- 管理 iteration budget
- 处理 fallback、retry、context compression
- 组织最终响应

可以把编排层理解成：

```text
Context -> LLM -> Tool Call -> Observation -> Context -> LLM -> Final
```

不应该负责：

- 具体平台协议
- 每个工具的内部实现细节
- 长期数据后端的具体存储实现

读源码时的定位方式：

```text
问题和模型调用、工具循环、上下文拼装、停止条件有关 -> 先看编排层
```

#### 5.3、能力层

能力层解决“**Agent** 能做什么”

它包含三个不同概念：

| 子系统 | 含义 |
|---|---|
| **Tools** | 当前可调用的外部动作，例如终端、文件、浏览器、子代理、MCP |
| **Skills** | 从经验中沉淀的过程性知识 |
| **Plugins** | 可替换或可扩展的能力后端 |

需要特别区分：

```text
Tool = 能做的动作
Skill = 怎么做的经验
Plugin = 能力后端的扩展方式
```

职责：

- 注册工具
- 描述工具 schema
- 控制工具是否暴露给模型
- 执行工具调用
- 管理技能检索和技能写入
- 接入外部能力系统

不应该负责：

- 决定整个任务是否结束
- 保存所有长期状态
- 处理用户所在平台协议

读源码时的定位方式：

```text
问题和文件读写、终端执行、浏览器、MCP、skill 创建有关 -> 先看能力层
```

#### 5.4、状态层

状态层解决“**Agent** 记住什么”

状态不是单一数据库概念，而是一组不同生命周期的数据：

| 状态类型 | 含义 |
|---|---|
| Session | 当前或历史会话 |
| Memory | 跨会话长期记忆 |
| Config | profile、provider、工具、路径等配置 |
| Logs | 执行过程和排查信息 |
| Jobs | 定时任务和后台任务状态 |

职责：

- 持久化 session
- 保存长期 memory
- 管理配置加载和覆盖
- 支持任务恢复和历史检索
- 为上下文拼装提供长期信息

风险：

- 错误记忆被长期使用
- 配置覆盖关系不清晰
- 会话状态和长期记忆混在一起
- 日志和私有数据边界不清

读源码时的定位方式：

```text
问题和 session、memory、config、logs、jobs 有关 -> 先看状态层
```

#### 5.5、平台层

平台层解决“**Agent** 如何到达用户”

它不是 **Agent** 内核，而是平台协议适配层

典型入口：

- **Telegram**
- **Discord**
- **Slack**
- Email
- REST API
- **ACP**

职责：

- 接收平台事件
- 标准化消息
- 映射 session
- 调用统一 **Agent Runtime**
- 把响应转换回平台消息

不应该负责：

- 决定模型如何推理
- 决定工具如何调度
- 直接管理长期记忆策略

读源码时的定位方式：

```text
问题和 Telegram、Discord、API、Webhook、ACP 有关 -> 先看平台层
```

### 六、五层架构的依赖关系与边界

- 重新生成的图能把重点收敛到“请求如何进入、核心如何编排、能力和状态如何被使用”

#### 6.1、五层架构边界图

![Hermes 五层架构边界图](assets/hermes-five-layer-boundary.png)

补充关系不画在连线上，避免图形变成复杂回路：

| 关系 | 说明 |
|---|---|
| 状态层 -> 编排层 | 状态层为编排层提供 session、memory、config 等长期上下文 |
| 能力层 -> 编排层 | 能力层执行工具后，把 observation 返回给编排层继续决策 |

#### 6.2、关键边界

| 边界 | 判断 |
|---|---|
| 入口层 vs 编排层 | 入口层负责接收请求，编排层负责组织 **Agent** 执行 |
| 平台层 vs 编排层 | 平台层负责协议转换，编排层不应该知道 **Telegram** 或 **Discord** 细节 |
| 编排层 vs 能力层 | 编排层决定何时调用工具，能力层负责工具具体执行 |
| 编排层 vs 状态层 | 编排层决定何时读写状态，状态层负责持久化和检索 |
| 能力层 vs 状态层 | 工具可以产生状态变化，但长期保存策略不能散落在工具内部 |

#### 6.3、为什么不是传统三层

传统三层通常是：

```text
表现层 -> 业务层 -> 数据层
```

但 **Hermes** 是 **Agent Runtime**，传统三层不足以表达：

- 多入口和多平台不是同一种表现层
- 工具和技能不是普通业务逻辑，而是模型可见的行动空间
- memory、session、skills、config 的生命周期不同
- 编排层要处理 **LLM**、工具、上下文、状态和停止条件的循环

五层架构的价值，是让读源码时先定位问题类型，而不是从目录树盲读

### 七、当前阶段需要记住的架构判断

#### 7.1、Hermes 的核心目标

**Hermes** 的核心目标不是单次对话质量，也不是单次编码任务效率

它优化的是：

```text
一个个人 Agent 和用户长期协作时的价值积累
```

#### 7.2、Learning Loop 是核心差异

如果去掉 **Learning Loop**，**Hermes** 会退化成更复杂的工具调用助手

**Skills**、**Memory Nudge**、**Background Review** 共同构成它的学习闭环

#### 7.3、CLI-First 是运行时边界选择

**CLI-First** 不只是交互偏好

它会影响：

- 配置如何存
- 状态如何持久化
- 工具如何执行
- 进程如何运行
- 远程平台如何投射

#### 7.4、五层架构是源码定位工具

后续遇到源码问题，先问：

| 问题 | 优先定位 |
|---|---|
| 请求从哪里进入 | 入口层 |
| **Agent** 如何决策和循环 | 编排层 |
| 能调用哪些工具或技能 | 能力层 |
| 什么信息被持久化 | 状态层 |
| 如何接入消息平台或 API | 平台层 |

#### 7.5、后续读源码的固定问题

后续读每一章，都可以用这几个问题做复习框架：

- 这个模块属于哪一层
- 它服务四个设计赌注中的哪一个
- 它处理的是输入、决策、能力、状态，还是平台适配
- 它的复杂度来自业务需求，还是来自长期运行和多环境约束
- 如果用 **JavaScript / TypeScript** 实现同类系统，边界是否相同
