## SQLite 相关

这篇笔记用于建立 **SQLite** 的核心心智模型，重点不是 **SQL** 语法，而是理解它为什么是本地数据库、它和普通文件的差异、它和 **MySQL** / **PostgreSQL** 这类服务端数据库的边界

### 一、核心定位：嵌入式关系型数据库

==**SQLite** 是一个嵌入式 **SQL** 数据库引擎，它以进程内工具库的形式被应用调用，不依赖独立数据库服务，直接读写本地数据库文件==

最短心智模型：

```text
普通文件：
应用自己管理数据结构和一致性

SQLite：
应用进程内嵌数据库引擎，由数据库引擎管理本地数据库文件

PostgreSQL / MySQL：
应用连接独立数据库服务，由服务端进程管理数据文件和多用户访问
```

所以 **SQLite** 可以记成：

```text
SQLite = 数据库引擎 + 本地数据库文件 + 文件锁 / 事务日志 / WAL
```

它的核心价值不是“把数据放进一个文件”，而是：

> 在没有独立数据库服务的前提下，提供真正的数据库语义，包括表、索引、约束、SQL、事务、一致性和崩溃恢复

### 二、为什么它是本地数据库

“本地数据库”不是因为系统默认自带 **SQLite**，而是因为它的运行方式天然偏本地

**SQLite** 的典型结构是：

```text
应用进程 -> SQLite 库 -> 本地数据库文件
```

常规服务端数据库的典型结构是：

```text
应用进程 -> 网络 / Socket -> 数据库服务进程 -> 数据文件
```

这意味着 **SQLite** 的数据库通常就是本机上的一个 `.db`、`.sqlite`、`.sqlite3` 文件，应用进程通过 **SQLite** 库直接读写它，不需要单独启动数据库服务器，也不需要配置端口、账号、网络连接和后台守护进程

有些系统确实会内置 **SQLite**，例如 **macOS** 通常带有 `sqlite3` 命令，**iOS**、**Android** 也广泛使用 **SQLite**，但这不是它被称为本地数据库的根本原因

根本原因是：

> **SQLite** 是嵌入式数据库，数据库引擎运行在应用进程里，数据落在本地文件中

如果要和浏览器环境类比，可以理解为：

```text
SQLite 之于本地应用
类似
IndexedDB 之于浏览器网页
```

但两者不是同一类数据库，**SQLite** 是完整的关系型数据库，使用 **SQL** 和关系模型，**IndexedDB** 更像浏览器提供的本地对象数据库，受浏览器同源策略和沙箱限制

### 三、和普通文件的本质区别：数据库引擎

**SQLite** 的底层确实是本地文件，但它不是普通文件存储

普通文件系统只负责保存字节：

```text
/path/app.db -> 一串字节
```

文件系统并不知道：

- 哪些字节代表 `users` 表
- 哪一列是 `email`
- 哪些字段必须唯一
- 哪些查询可以走索引

如果用 **JSON**、**CSV**、**Excel** 存数据，通常是应用代码自己解释结构、自己过滤、自己写回：

```text
应用代码 -> 读取整个文件 -> 自己解析 -> 自己过滤 -> 自己写回
```

**SQLite** 则是：

```text
应用代码 -> SQL -> SQLite 引擎 -> 页、B-Tree、索引、事务日志 -> 数据库文件
```

中间这层数据库引擎负责：

- 表结构和元数据管理
- **schema**、主键、唯一约束、外键和检查约束
- **SQL** 查询和查询优化
- 事务、锁和崩溃恢复

所以 **SQLite** 不是“普通文件 + 表格结构 + SQL 查询”，而是把数据库引擎封装进应用进程，再由数据库引擎管理本地文件

这也是它和 **Excel** 的根本区别：

> **Excel** 让人方便地看表和改表，**SQLite** 让程序可靠地管理结构化数据

### 四、和服务端数据库的本质区别：少了中央服务

**SQLite** 和 **MySQL** / **PostgreSQL** 的主要差异不是“能不能用 SQL”，而是中间有没有独立数据库服务

服务端数据库：

```text
多个应用 -> 数据库服务进程 -> 数据文件
```

**SQLite**：

```text
多个应用进程 -> 各自内嵌 SQLite 引擎 -> 同一个数据库文件
```

常规数据库的独立服务进程通常负责：

- 连接管理
- 权限校验
- 事务调度
- 锁管理
- 并发控制
- 查询执行和优化
- 数据文件读写
- 复制、备份、高可用、监控等服务端能力

**SQLite** 没有这个中央服务进程，每个应用进程里都有自己的 **SQLite** 引擎实例，多个进程访问同一个 `.db` 文件时，需要通过文件锁、事务日志、**WAL** 等机制协调读写

可以把两类数据库记成：

```text
PostgreSQL / MySQL：
大家先去数据库服务排队，由服务统一验权、调度和写文件

SQLite：
大家各自带着 SQLite 引擎，直接去同一个文件前协调读写
```

重点不是 **SQLite** 没有数据库能力，而是它把数据库能力放进应用进程里，而不是放在一个独立服务端进程里

### 五、事务、锁、权限与并发边界

理解 **SQLite** 的并发边界，需要先把事务、锁、权限拆开

事务解决的是：

> 一组相关操作，不能只成功一半

例如转账：

```sql
begin;

update accounts set balance = balance - 100 where id = 'A';
update accounts set balance = balance + 100 where id = 'B';

commit;
```

结果只应该有两种：

- 两条更新都成功
- 两条更新都不生效

锁解决的是：

> 多个人同时改同一份数据时，谁先谁后

例如库存只有 `1` 件时，如果两个事务同时读取库存并同时下单，可能出现超卖，锁的作用就是让冲突修改按顺序发生，或者让后一个事务等待、失败、重试

权限解决的是：

> 谁能对哪些数据做哪些操作

常规数据库有服务端用户和权限体系，**SQLite** 不内置服务端式的多用户权限管理，它更多依赖操作系统文件权限和应用层权限控制

因此 **SQLite** 不是不支持并发，而是不适合高并发写入：

- 多个读可以并行
- 一个写事务提交时，其他写通常需要等待
- 开启 **WAL** 模式后，读写并行体验会更好
- 写入很多、写冲突很多时，容易出现排队、锁等待、超时或吞吐下降

可以记成：

```text
多个读：可以并行
多个写：需要排队
读 + 写：WAL 模式下体验更好
```

它的瓶颈不在于“不会事务”或“不会锁”，而在于：

> 它没有服务端数据库那种集中连接治理、权限校验和更细粒度的大规模并发调度能力

### 六、适用场景与最终记忆锚点

**SQLite** 适合：

- 桌面软件
- 移动应用
- **CLI** 工具
- 本地开发工具
- **Electron** 应用
- 单机小型服务
- 读多写少的应用
- 测试环境
- 需要简单部署和单文件备份的场景

**SQLite** 不适合：

- 多服务共同高频写入
- 多用户中心化数据库
- 复杂服务端权限治理
- 横向扩展和高可用集群
- 高频写入且写冲突严重的业务
- 需要数据库服务统一治理连接、审计、复制、备份和运维的系统

一句话判断：

```text
单机、本地、读多写少、写入频率可控：适合 SQLite

多服务、多用户、高频写入、需要集中权限治理和横向扩展：优先考虑 PostgreSQL / MySQL
```

最终可以用四句话记住 **SQLite**：

> 文件系统保存字节

> **Excel** 给人编辑表格

> **SQLite** 给程序在本地管理结构化数据

> **PostgreSQL** / **MySQL** 给多用户、多服务集中管理结构化数据

再压缩成一句：

> **SQLite** 是“带数据库引擎的本地文件”，不是“带本地文件的数据库服务”
