## 流式传输

### 一、 概述

#### 1.1、什么是流式传输

**流式传输（Streaming）** 是一种将数据切割成小块，并在数据下载的同时进行实时处理或展示的技术，而不是等待整个文件完全下载完毕后再使用

**核心特点：**

- **实时性**：用户无需等待完整下载，可即时访问分块内容
- **内存友好**：只需在内存中保留当前处理的数据块，非常适合处理 GB 级别的大文件或持续不断的长期数据

#### 1.2、流式传输常见的实现方式

- **HTTP 流式传输 (Fetch + ReadableStream)**：通过 `fetch` 请求获取数据流，前端利用 `ReadableStream` 逐块读取数据
  
  - **HTTP/1.1** 协议允许服务器不发送 `Content-Length`，而是将数据切分成多个“块（**Chunk**）”按需发送
  
    > 核心机制是： **HTTP** 的 `Transfer-Encoding: chunked` 特性，仅将响应视为一个“字节管道”
    >
    > 配置 `Transfer-Encoding: chunked` 仅仅是告诉接收方：“我要开始按块发数据了，别等 **Content-Length** 了
  
  - ==浏览器不需要遵循任何特殊格式，需要开发者在代码中定义数据边界==
  
- **SSE (Server-Sent Events)**：一种允许服务器主动向客户端推送数据的**单向通信**技术
  
  - 规定服务器必须按照 `data: ...\n\n` 的特定格式发送数据
  - ==浏览器提供 **EventSource 接口**原生支持自动重连和消息解析==
  
- **WebSocket**：支持**全双工通信**，适合需要客户端与服务端频繁双向交互的场景

- **WebRTC**：点对点（**P2P**）传输技术，无需经过服务器中转，适合极低延迟的视频聊天或文件传输

#### 1.3、流式传输中常见的问题

在通过网络进行**流式传输**时，数据包在传输过程中会面临以下问题：

- **分包顺序问题 **：数据包可能会乱序到达 <!--无需关注，TCP 层自动解决了，保证客户端拿到的数据流，一定是按顺序的-->
- **粘包问题 **：发送方发送了两个独立的消息（例如消息 A 和消息 B）接收方可能一次性收到了 `A + B` 的合并数据包，导致无法区分哪部分属于 A，哪部分属于 B
- **拆包问题 **：发送方发送的一条完整消息，可能被切分成多次到达，接收方只收到 A 的前半部分，导致无法立即解析成完整对象

### 二、Raw HTTP Streaming (Fetch 流)

**Node.js 后端示例**

```JavaScript
const express = require('express');
const app = express();

app.get('/raw-stream', (req, res) => {
    // 1. 设置正确的 Header
    // text/plain 或 application/octet-stream 均可
    res.setHeader('Content-Type', 'text/plain; charset=utf-8');
    
    // 2. 模拟一个耗时的数据生成过程
    let count = 0;
    const timer = setInterval(() => {
      // 直接写入原始数据
      // res.write() 会自动添加 Transfer-Encoding: chunked 响应头
      // 正常使用 res.send() 或 res.json()时，会先将所有数据在内存中攒齐（Buffer），计算出数据的总大小
      // 然后自动设置 Content-Length 响应头，最后一次性发送给客户端
      // 浏览器在收到完整的 Content-Length 大小之前，会一直处于“正在加载”状态
        res.write(`这是第 ${++count} 个数据块\n`);
        
        if (count >= 5) {
            res.write('流传输结束');
            res.end(); // 必须调用 end，否则客户端会一直等待
            clearInterval(timer);
        }
    }, 1000);
});

app.listen(3000, () => console.log('Raw Stream Server: http://localhost:3000/raw-stream'));
```

**前端实现 (Fetch)**

```JavaScript
async function consumeStream() {
    const response = await fetch('http://localhost:3000/raw-stream');
    
    // 获取可读流的 reader
    const reader = response.body.getReader();
    const decoder = new TextDecoder(); // 用于将二进制流转换为字符串

    while (true) {
        // 读取数据块
        const { done, value } = await reader.read();
        
        if (done) {
            console.log('流已全部读取完毕');
            break;
        }

        // 解码并打印数据块
        const chunk = decoder.decode(value, { stream: true });
        console.log('收到新数据块:', chunk);
    }
}

consumeStream();
```

**Raw HTTP Streaming 的优缺点**

- **优点**：
  - 可以利用 **POST** 发送复杂上下文
  - 可以在响应中自定义任何二进制协议
  - 没有任何冗余的 `data:` 前缀，带宽利用率更高
- **缺点**：
  - 需自己处理 `TextDecoder` 来防止中文字符乱码
  - 需要自己解决**「粘包问题」**和**「拆包问题」**
    - 由于没有内置的格式来标记哪里是结尾
    - 须自己实现一种边界协议（例如：固定长度法、特定结束符法）
    - 须在前端维护一个全局 `buffer`，每收到一个 `chunk` 就必须执行一次“是否包含完整消息”的逻辑判断

  - 需自己处理断线问题


### 三、SSE (Server-Sent Events)

**Node.js 后端示例 (Express)**

```JavaScript
// server.js
const express = require('express');
const app = express();

app.get('/api/sse', (req, res) => {
    // 1. 设置响应头，声明 SSE 类型
    res.setHeader('Content-Type', 'text/event-stream');
    res.setHeader('Cache-Control', 'no-cache');
    res.setHeader('Connection', 'keep-alive');
    res.setHeader('Access-Control-Allow-Origin', '*'); // 允许跨域

    // 2. 发送一个初始连接成功的消息
    res.write(`data: ${JSON.stringify({ message: '连接已建立' })}\n\n`);

    // 3. 定时推送数据
    const intervalId = setInterval(() => {
        const data = {
            time: new Date().toLocaleTimeString(),
            value: Math.floor(Math.random() * 100)
        };
        
        // 注意：SSE 格式必须以 'data: ' 开头，以 '\n\n' 结尾
        res.write(`data: ${JSON.stringify(data)}\n\n`);
    }, 2000);

    // 4. 当客户端断开连接时，清理定时器
    req.on('close', () => {
        clearInterval(intervalId);
        res.end();
    });
});

app.listen(3000, () => {
    console.log('SSE Server running on http://localhost:3000');
});
```

**前端实现**

```JavaScript
// 前端无需安装任何库，直接使用浏览器内置的 EventSource 对象即可

// 1. 创建 EventSource 实例，连接到后端 SSE 接口
const eventSource = new EventSource('http://localhost:3000/api/sse');

// 2. 监听默认消息事件
eventSource.onmessage = (event) => {
    const data = JSON.parse(event.data);
    console.log('收到数据:', data);

    document.getElementById('display').innerHTML = `
        <p>时间: ${data.time || 'N/A'}</p>
        <p>数值: ${data.value || data.message}</p>
    `;
};

// 3. 监听连接打开
eventSource.onopen = () => {
    console.log('连接已打开');
};

// 4. 监听错误
eventSource.onerror = (err) => {
    console.error('连接出现错误:', err);
};
```

**SSE 的优缺点**

- **优点**：
  - 代码极简，浏览器处理好了异常断线后的重连，极其稳定
  - **SSE** 通过协议层面的约定，完美解决了**「粘包问题」**和**「拆包问题」**
    - **SSE** 强制规定数据以 `data: <内容>\n\n` 的格式传输
    - **浏览器内核 (EventSource)** 充当了“智能解析器”
      - 它不关心收到的是半个包（拆包）还是两个包挤在一起（粘包），它只认 `\n\n` 这个**分界符**
      - 它会维护一个内部缓冲区，把收到的数据拼起来，直到检测到 `\n\n`，它才确信这部分是一个完整的消息，并将其作为一次独立的 `onmessage` 事件从而调用注册好的监听函数

- **缺点**：
  - 受限于 **GET** 请求，无法携带复杂的 **Body** 参数
  - 协议格式僵化，不适合传输结构复杂的业务数据
