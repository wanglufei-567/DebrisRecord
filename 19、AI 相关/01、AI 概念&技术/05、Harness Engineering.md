## Harness Engineering

### 一、Harness Engineering 是什么

**Harness Engineering** 是围绕 **LLM Agent** 设计的外部工程系统，让模型在**可观测**、**可验证**、**可回滚**、**可持续改进**的边界内行动

它解决的不是“如何写好一句 **Prompt**”，而是：

- **Agent** 能看到什么上下文
- **Agent** 能调用什么工具
- **Agent** 能访问哪些数据、修改哪些文件
- **Agent** 如何记录任务状态
- **Agent** 如何验证结果
- 失败后如何重试、恢复、回滚
- 哪些动作必须经过人工批准
- 哪些规则由测试、类型、权限、沙箱强制执行

更直观地说：

> ==**Agent = Model + Harness**==
>
> ==**Model** 提供推理、生成和规划能力==
>
> ==**Harness** 提供工具、上下文、状态、权限、验证、反馈和执行边界==

**Harness Engineering** 的核心不是训练一个更听话的模型，而是设计一个让模型难以越界、容易纠错、可以持续产出的工程系统

**Harness** 原意是“约束装置”，在 **AI Agent** 中指包裹在模型外面的运行系统

一个完整的 **Harness** 通常包含：

- **指令层**：系统提示词、任务模板、`AGENTS.md`、技能说明、编码规范
- **上下文层**：**RAG**、项目文档、长期记忆、任务状态、上下文压缩
- **工具层**：文件系统、终端、浏览器、数据库、**GitHub**、**MCP**、业务 **API**
- **执行层**：任务循环、工具调度、子任务拆分、重试、超时、队列
- **验证层**：单元测试、类型检查、**Lint**、端到端测试、截图验证、评估集
- **观测层**：日志、指标、**Trace**、工具调用轨迹、失败样本、成本统计
- **安全层**：沙箱、权限审批、网络隔离、文件边界、敏感操作阻断
- **治理层**：**PR** 流程、人工审核、技术债清理、规则沉淀

所以，**Harness Engineering** 更接近“运行时 + 平台工程 + 测试体系 + 安全边界 + 可观测性”的组合，只是它服务的对象从确定性程序变成了非确定性的 **LLM Agent**

### 二、与 Prompt Engineering、Context Engineering 的关系

三者不是替代关系，而是逐层扩大：

- **Prompt Engineering**：语言控制，关注“怎么对模型说”
- **Context Engineering**：信息控制，关注“让模型看到什么”
- **Harness Engineering**：系统控制，关注“模型在什么环境中行动”

可以简化为：

```text
Prompt Engineering  ⊂  Context Engineering  ⊂  Harness Engineering
```

区别在于：

- **Prompt** 可以约束表达，但不能可靠约束行为
- **Context** 可以补充信息，但不能保证执行正确
- **Harness** 用工具、权限、状态、测试、观测和反馈，把模型行为纳入工程闭环

越往后，越接近软件工程；越往前，越接近模型交互技巧

### 三、最小实现闭环

可以把 **Harness Engineering** 的落地抽象为三段闭环：

```text
上下文管理 → 验证与反馈 → 技术债清理 → 反向更新上下文与规则
```

这三个环节不是 **Harness** 的全部组件，但构成了最小可用闭环

#### 1. 上下文管理

目标：让 **Agent** 在行动前看到正确、足够、不过载的信息

典型做法：

- 用 `AGENTS.md` 提供仓库级入口规则
- 用 `docs/` 存放架构、产品规格、**API** 约定、编码规范
- 用任务计划、进度文件、**Git** 历史保存跨会话状态
- 用 **RAG** 或检索工具按需注入代码和文档
- 对长任务做上下文压缩，避免每轮塞入全部历史

判断标准：**Agent** 不需要反复猜项目结构，也不会因为上下文过多而忽略关键约束

#### 2. 验证与反馈

目标：把 **Agent** 的输出转化为可判定、可追踪、可修复的反馈信号

典型做法：

- 单元测试、类型检查、**Lint**、构建检查
- 端到端测试、浏览器自动化、截图验证
- 日志、指标、**Trace**、错误堆栈、工具调用记录
- CI 质量门禁、**PR Review**、人工审批
- 把失败信息写成 **Agent** 能理解的修复指令

判断标准：**Agent** 不能只说“完成了”，必须用测试、日志、截图、指标或审查结果证明完成

#### 3. 技术债清理

目标：把验证阶段暴露的问题沉淀为结构改进，而不是只修一次表面错误

典型做法：

- 将重复错误写入规则、文档、测试或 **Lint**
- 清理 **Agent** 生成的重复代码、临时文件、过大函数、过度抽象
- 建立架构边界测试，防止模块依赖持续劣化
- 复盘失败样本，更新 **Prompt**、上下文组织、工具描述和评估集
- 定期安排后台 **Agent** 或人工任务处理技术债

判断标准：同类错误下一次更难发生，系统不会因为 **Agent** 的高产出而快速积累混乱

技术债清理尤其要关注“依赖方向”，例如编码 **Agent** 场景中常见的分层：

```text
UI → Runtime → Service → Repo → Config → Types
```

合理方向是上层依赖下层，不能让下层反向感知上层

- 例如 **UI** 可以调用 **Runtime** 或 **Service**，**Service** 可以调用 **Repo**，但 **Repo** 不应该感知页面交互，**Types** 不应该依赖业务实现

因此，**Harness** 需要把架构边界变成可验证规则：

- 用架构测试或 **Lint** 禁止反向依赖
- 用模块边界规则限制跨层 **import**
- 用类型契约固定层间输入输出
- 在 **CI** 和 **Review** 中检查 UI 修复是否污染 **Service**、**Repo**、**Types**
- 把常见违规模式写进 `AGENTS.md` 或仓库规则

这套闭环的关键是：

- **上下文管理** 决定 **Agent** 做事前知道什么
- **验证与反馈** 决定 **Agent** 做事后如何知道对错
- **技术债清理** 决定系统能否把一次错误转化为长期改进

如果只做上下文管理，**Agent** 仍可能自信地做错

如果只做验证，团队会反复修同类问题

如果只做技术债清理，但没有上下文管理，**Agent** 仍会在错误信息结构里工作

### 四、OpenAI 的实战：Agent-first 软件工程

**OpenAI** 在 [Harness engineering: leveraging Codex in an agent-first world](https://openai.com/index/harness-engineering/) 中，描述了一个内部实验：小团队用 **Codex** 构建并交付一个内部产品，代码、测试、CI、文档、可观测性和内部工具都由 **Codex** 生成，人类不直接手写代码

这个案例的重点不是“模型很强”，而是 **OpenAI** 把仓库、应用、验证、观测和治理都改造成 **Agent** 可理解、可执行、可反馈的环境

<img src="https://raw.githubusercontent.com/wanglufei561/picture_repo/master/assets/openai-agent-first-harness.png" alt="openai-agent-first-harness" style="zoom: 25%;" />

**OpenAI** 的思路可以概括为四点：

- **人类从写代码转向设计环境**
  - 人类负责意图、任务边界、反馈环路和规则沉淀
  - **Codex** 负责执行、改代码、跑验证、发 PR、响应反馈

- **让仓库成为 Agent 的事实来源**
  - `AGENTS.md` 作为入口目录，而不是巨大的说明书
  - 深层知识放进结构化 `docs/`，包括架构、产品规格、质量规则、可靠性要求
  - 重要决策、执行计划、技术债进入版本控制，减少隐性上下文

- **让应用对 Agent 可读**
  - 每个工作区能启动独立应用实例
  - **Agent** 通过浏览器自动化、**DOM**、截图、日志、指标和 **Trace** 观察系统
  - 这样它不只是读代码，而是能复现问题、验证修复、观察运行状态

- **用机械规则约束架构**
  - 自定义 **Lint**、结构测试、分层依赖规则、命名规则、文件大小限制进入 **CI**
  - 规则不是建议，而是强制边界
  - 错误信息要写成 Agent 能理解的修复指令

**OpenAI** 的 **Harness** 可以总结为一个闭环：

```text
人类意图 → Codex 执行 → 测试 / 观测 / Review → 规则沉淀 → 仓库上下文更新 → 下一轮 Codex 执行
```

这个闭环的关键是：每一次失败都不只修当前 bug，还要尽量转化为下一次 **Agent** 更难犯错的环境改造

因此，**OpenAI** 的实践更像是 **Agent-first 软件工程**：

- 仓库不只是给人看的，也是给 **Agent** 读的
- 日志、指标、**Trace** 不只是给运维看的，也是给 **Agent** 调试用的
- **Lint**、**CI**、架构规则不只是约束人，也是约束 **Agent** 的机械边界
- 技术债不只是 **Review** 产物，也要进入 **Agent** 可持续清理的任务系统

当代码产出速度由 **Agent** 放大后，瓶颈从“谁来写代码”转向“谁来定义边界、验证结果、清理熵、保持系统可维护”

### 五、Anthropic 的实战：长任务 Agent Harness

**Anthropic** 也有两篇文章：

- [Effective harnesses for long-running agents](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents) 先解决“长任务如何跨上下文窗口持续推进”
- [Harness design for long-running application development](https://www.anthropic.com/engineering/harness-design-long-running-apps) 进一步解决“任务推进后，如何避免需求误解、自评偏宽和实现质量失控”

这两篇文章不是简单重复同一个主题：第一篇处理长任务的**状态连续性**，第二篇处理状态连续之后暴露出的**质量判断问题**

第一步让 **Agent** 能“接着做”，第二步让 **Agent** 能“按正确标准做”

#### 1. 跨会话交接：Initializer + Coding Agent

[Effective harnesses for long-running agents](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents) 讨论的是长任务的基础问题：上下文窗口有限，每个新会话都可能像“失忆的新工程师”一样重新开始

**Anthropic** 观察到的典型失败模式：

- **Agent** 一次性做太多，导致上下文耗尽
- 前一个会话留下半成品，后一个会话不知道发生了什么
- 后续 **Agent** 看到已有进展后过早宣布完成
- **Agent** 修改了代码，但没有做真正的端到端验证

对应的 **Harness** 设计：

- **Initializer Agent**
  - 第一个会话负责搭建环境
  - 创建 `init.sh`、进度文件、结构化功能清单、初始 Git 提交
  - 把大目标拆成可逐步验证的功能项

- **Coding Agent**
  - 后续会话每次只推进一个功能点
  - 开始前读取进度文件和 **Git** 历史
  - 结束时提交代码、更新进度、留下可交接状态

- **结构化验收**
  - 功能清单使用 **JSON** 维护，每个功能有明确步骤和 `passes` 状态
  - **Agent** 只能在充分测试后把功能标记为通过
  - 相比 **Markdown**，**JSON** 更不容易被模型随意改写

- **端到端验证**
  - **Web** 应用要用浏览器自动化像真实用户一样验证
  - 只运行单元测试或 `curl` 不足以证明功能可用

这套设计的本质是：把上下文窗口里的隐性状态，转成下一轮 **Agent** 可读取的显式事实

#### 2. 分阶段控制：Planner + Generator + Evaluator

[Harness design for long-running application development](https://www.anthropic.com/engineering/harness-design-long-running-apps) 是在第一步基础上的升级：有了进度文件、功能清单和端到端验证后，仍然会出现需求理解偏差、自我评估偏宽、实现质量不稳定

**Anthropic** 将任务拆成三个角色：

- **Planner**：拆解需求，生成功能清单、验收标准和初始环境
- **Generator**：选择功能点，提出验收标准，生成代码并提交实现
- **Evaluator**：检查验收标准与实现结果，给出修改建议或评估反馈

<img src="https://raw.githubusercontent.com/wanglufei561/picture_repo/master/assets/anthropic-long-running-harness.png" alt="anthropic-long-running-harness" style="zoom:25%;" />

这个结构包含两个关键闭环：

- **验收标准闭环**
  - **Generator** 先提出“这个功能怎样才算完成”
  - **Evaluator** 检查验收标准是否完整、可测试、可执行
  - 标准不充分时，**Evaluator** 反馈修改建议，**Generator** 继续修正

- **实现结果闭环**
  - **Generator** 按通过后的验收标准生成代码
  - **Evaluator** 用测试、运行结果、浏览器操作、日志或人工规则评估实现
  - 实现不通过时，**Evaluator** 返回具体反馈，**Generator** 继续修改

这比“让一个 **Agent** 一口气从需求写到代码”更可靠，因为它把任务拆成两个可验证阶段：先验证目标是否清楚，再验证实现是否正确

更重要的是，**Evaluator** 是外部质量判断者，避免 **Generator** 自己判断自己是否完成

这能降低长任务 **Agent** 的常见风险：模型投入大量上下文和行动成本后，容易倾向于认为“差不多完成了”

所以，Anthropic 的演进不是“多设计几个 Agent 角色”，而是从“解决上下文断裂”推进到“解决质量判断断裂”

- 第一篇的核心是 **状态连续性**
- 第二篇的核心是 **评估独立性**
- 两者共同目标是让长任务 Agent 从“能继续做”走向“能可靠做完”

#### 3. 运行时边界：Session / Harness / Sandbox

**Anthropic** 的 **Managed Agents** 实践进一步把系统拆成三个接口：

- **Session**：追加式记录整个任务发生过什么
- **Harness**：调用 **Claude**、路由工具调用、管理循环
- **Sandbox**：真正运行代码、编辑文件、执行命令的环境

这个拆分把推理、工具执行、状态记录、权限隔离分开，系统更容易观测、复现和回滚

**Anthropic** 还有一个重要判断：**Harness 不是越复杂越好，而是要把模型当前不稳定的地方外置成工程约束**

如果模型已经能稳定完成某个步骤，**Harness** 应该减少流程；如果模型经常在某个步骤失败，**Harness** 就应该把它变成显式状态、显式验收或显式工具反馈

### 六、Harness 来源

**Harness Engineering** 不是某一篇论文或某家公司发明的新算法，而是许多公司在 **Agent** 实践落地后的默契命名：

- **Test Harness**：测试工程中早已有“测试支架”，用于注入输入、捕获输出、判断结果
- **Evaluation Harness**：LLM 评测框架会统一加载数据集、调用模型、计算指标
- **Agent 工程**：当模型开始调用工具、修改文件、操作浏览器、跨步骤执行任务，外部系统就从辅助设施变成可靠性的核心

结论：**Harness Engineering 有概念包装成分，但不是空洞噱头；它不是 Transformer 级别的底层技术突破，而是 Agent 时代的软件工程范式变化**

说它不是纯噱头，是因为它解决的是真问题：

- **LLM** 是概率系统，输出不可完全预测
- **Agent** 会跨工具、跨文件、跨会话执行任务
- **Prompt** 无法可靠约束所有行为
- 上下文窗口无法承载所有知识和历史
- **生产系统需要权限、审计、回滚、验证和可观测性**

说它不是底层技术突破，是因为它没有提出新的模型结构，也没有从根本上消除幻觉、推理错误或安全风险

很多 **Harness** 实践本来就是成熟软件工程：

- 自动化测试
- CI/CD
- 类型系统
- Lint
- 沙箱权限控制
- 日志与监控
- 代码评审
- 可回滚变更

这些工程手段过去主要约束人写的代码，现在要系统性地约束 **AI Agent** 的**感知**、**行动**和**反馈**循环

如果涉及**长任务**、**自动改代码**、**自动操作系统**、**访问私有数据**、**执行生产流程**，就必须考虑 **Harness**

### 七、总结

==真正可用的 **Agent**，不只是一个更聪明的模型，而是一个被**正确约束**、**正确供给上下文**、**正确验证结果**的系统==

**Harness Engineering 是当下阶段的工程解法**：它把模型暂时**不稳定**、**不可靠**、**不可控**的部分，外置成**可验证的工程约束**

**Harness Engineering** 会随着模型变强而变薄，但不会简单消失

从技术演进规律看，很多工程层都会经历类似过程：

- 早期能力不足时，外围系统承担大量补偿职责
- 核心能力增强后，一部分外围逻辑被内化、合并或自动化，但承担安全、权限、审计、边界的系统层不会消失

模型能力越弱，**Harness** 越厚：

- 需要更详细的 **Prompt**
- 需要更多结构化上下文
- 需要更细的任务拆解
- 需要更多检查点、评估器和人工审批
- 需要更严格的沙箱和权限边界

模型能力越强，**Harness** 会变薄：

- 一些显式流程会被模型内化
- 一些固定规则会变成模型默认能力
- 一些中间 **Agent** 角色会合并
- 一些低风险任务可以减少人工审批
- 上下文压缩、计划维护、错误恢复会更自动化

但 **Harness** 不会完全消失，因为它还有一部分职责不是“补模型短板”，而是“承载工程责任”：

- 权限控制不能只靠模型自觉
- 审计和回滚必须由系统保证
- 生产数据访问必须有隔离和授权
- 业务规则、合规要求、架构边界需要可验证机制
- 高风险动作仍需要人工或确定性系统兜底

**未来 Harness 中“补偿模型能力不足”的部分会减少，但“承载工程责任、权限边界、验证闭环”的部分会长期存在**
