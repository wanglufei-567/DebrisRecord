## Codex 使用指南

### 一、基础结构认知

#### 1.1、项目与对话的关键概念

使用 **Codex** 时，可以先记住这条主线：**Codex** 会在一个项目范围内，围绕一段对话持续处理用户给出的任务

几个关键概念可以这样理解：

- **Project / Workspace**：文件和运行边界，决定 **Codex** 在哪个项目里工作
- **Thread**：一段持续对话，同一个 **Thread** 里的后续消息通常会沿用已有上下文
- **Context**：当前可见上下文，包括对话、读过的文件、工具结果和必要摘要
- **Goal**：可选的持续目标，挂在 **Thread** 上，可以跨多个执行回合推进
- **Plan**：可选的执行计划，用来说明接下来准备怎么做
- **Turn**：一次执行回合，通常由一次用户任务触发
- **Item**：**Turn** 里的具体事件，例如消息、工具调用、命令、文件改动、审批、回复
- **Tool Call / Process**：外部执行，例如读写文件、运行命令、使用浏览器；命令真正跑起来后就是具体 **Process**
- **Compaction**：长线程的上下文压缩机制，用来在对话变长后保留关键信息

几个常见操作可以这样判断：

- 想补充约束或修正方向：直接在当前 **Thread** 里追加消息
- 想临时挂起持续目标：暂停 **Goal**
- 想马上阻止当前执行：停止当前执行回合
- 想换一个完全独立任务：新开 **Thread**
- 想基于当前进展继续：留在当前 **Thread** 继续发消息

完整结构可以这样记：

```text
Project / Workspace：文件和运行边界
└─ Thread：一段持续对话
   ├─ Context：当前可见上下文
   ├─ Goal：可选的持续目标
   ├─ Plan：可选的执行计划
   ├─ Turn：一次执行回合
   │  ├─ Item：消息、工具、命令、文件改动、审批、回复
   │  └─ Tool Call / Process：外部执行
   └─ Compaction：长线程的上下文压缩机制
```
