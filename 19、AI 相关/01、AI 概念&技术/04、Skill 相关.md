## Skill 相关

### 一、Skill 是什么

**Skill** 中文意思是“技能”，技能是指一个人掌握了做某一类事情的标准流程

例如，一个人具备做饭的技能，不是说他认识锅、菜刀和食材，而是掌握了整套做饭的标准流程：

- 什么时候备菜
- 哪些食材先处理
- 哪些步骤不能颠倒
- 需要哪些工具
- 最后怎么判断菜是否做好

放到 **Agent** 系统里，**Skill** 也是类似的概念

它不是给模型增加一个新大脑，也不是直接提供某个工具，而是把“完成一类任务的方法”写成 **Agent** 能读懂、能复用的说明

通俗地说：

> **Skill 是给 Agent 使用的专项任务说明书**

工程化地说：

> **Skill 是把“如何完成一类任务”封装成 Agent 可调用、可复用、可维护的能力单元**

#### 1.1、最简单的 Skill 长什么样

一个最小的 **Skill** 可以很简单，本质上就是一段可复用的 **Prompt** 或说明文档

例如一个“**代码 Review Skil**l”可以这样写：

```md
---
name: code-review
description: 当用户要求 review 代码、检查 PR、找 bug 时使用
---

## 工作流程

1. 先阅读变更内容，理解改动目标
2. 优先检查 bug、行为回归、安全风险和缺失测试
3. 按严重程度排序问题
4. 每个问题都要给出文件位置、原因和建议
5. 如果没有发现问题，明确说明没有发现阻塞风险
```

这已经是一个 **Skill** 的雏形

它有触发条件，也有任务流程，能让 **Agent** 在代码 **Review** 场景下按固定方法工作

所以，**Skill** 的起点并不复杂，可以先理解为“**沉淀下来的一段任务 Prompt**”

只是工程化之后，**Skill** 往往不会只停留在一段 **Prompt** 上，它还可以继续组织脚本、模板、参考资料和素材，让复杂任务也能按步骤完成

#### 1.2、Skill 的核心组成

从最小形态看，**Skill** 至少要回答三个问题：

```text
Skill = 触发条件 + 操作流程 + 外部资源
```

- **触发条件**：什么时候应该使用这个 **Skill**
- **操作流程**：这类任务应该按什么步骤做
- **外部资源**：执行时可能需要哪些脚本、资料、模板或素材

### 二、Skill 的分工定位

**Agent** 由不同的技术构成：**LLM**、**Agent**、**Tool**、**Runtime**、 **Sandbox**、**Skill**  等

它们的分工可以这样看：

| 概念 | 作用 |
| ---- | ---- |
| **LLM** | 负责理解、推理、生成、规划和决策 |
| **Agent** | 围绕目标组织上下文、选择工具、执行步骤并根据反馈继续调整 |
| **Tool** | 外部能力接口，例如文件系统、终端、浏览器、数据库、MCP 工具 |
| **Runtime** | 代码、命令、脚本或任务实际运行的环境 |
| **Sandbox** | 限制文件、网络、命令、权限等资源访问的隔离边界 |
| **Skill** | 告诉 Agent 如何完成一类任务，以及何时读取说明、调用工具和加载资源 |

继续用“做饭”类比：

| 类比对象 | 在 Agent 系统中对应什么 |
| -------- | ----------------------- |
| 会理解和判断的人 | **LLM** |
| 负责完成整件事的人 | **Agent** |
| 锅、刀、烤箱 | **Tool** |
| 厨房 | **Runtime** |
| 厨房安全规则 | **Sandbox** |
| 菜谱和操作经验 | **Skill** |

其中

- **Tool** 解决的是“能做什么动作”
  - 例如读取文件、运行命令、打开浏览器

- **Skill** 解决的是“如何把这些动作组织成一套任务流程”
  - 例如创建文档时要先生成内容，再渲染预览，再检查排版，最后修正问题

### 三、Skill 的三层逻辑架构

一个完整的 **Skill** 目录示例：

```text
documents/
├── SKILL.md
├── scripts/
│   └── render_docx.py
├── references/
│   └── layout-rules.md
└── assets/
    └── report-template.docx
```

这个 **Skill** 用来处理 `.docx` 文档

它不是把所有内容都塞进一个超长 **Prompt**，而是通过 `SKILL.md`、`scripts`、`references`、`assets` 组织成一个能力模块

需要注意的是：**三层架构是逻辑分层，不是说每一层都必须对应一个独立文件**

例如，识别层的元信息通常写在 `SKILL.md` 顶部，指令层的 **SOP** 也写在 `SKILL.md` 正文里

| 逻辑层 | 主要承载位置 | 模块化内容 | 核心作用 |
| ------ | ------------ | ---------- | -------- |
| 识别层 | `SKILL.md` 顶部元信息 | `name`、`description`、触发条件、适用边界 | 判断什么时候启用这个 Skill |
| 指令层 | `SKILL.md` 正文 | SOP、行为约束、工具规则、按需加载策略、质量要求 | 定义任务应该怎么做 |
| 资源层 | `scripts`、`references`、`assets` | 脚本、参考资料、模板、素材 | 在具体步骤触发时提供执行材料 |

#### 3.1、识别层：`SKILL.md` 中的元信息

识别层负责让 **Agent** 判断“当前任务是否应该进入这个能力模块”

它通常不是单独放在另一个地方，而是写在 `SKILL.md` 的头部元信息或描述区域中

例如：

```md
---
name: documents
description: 当用户需要创建、编辑、渲染或检查 docx 文档时使用
---
```

它通常包含：

- **Skill 名称**：表达这个能力模块的主题
- **Skill 描述**：说明它解决哪类问题
- **触发条件**：说明用户请求中出现什么意图时应该使用
- **适用边界**：说明什么时候不该使用，避免误触发

从编程类比看，识别层像模块声明、路由规则或函数签名

它不负责真正执行任务，只负责告诉 **Agent**：这个请求应该交给哪个能力模块处理

#### 3.2、指令层：`SKILL.md` 中的 SOP

指令层是 **Skill** 的主逻辑，也通常写在 `SKILL.md` 里

同一个 `SKILL.md` 可以同时包含“元信息”和“操作手册”：

- 顶部元信息服务于识别层
- 正文 **SOP** 服务于指令层

`SKILL.md` 不是普通说明文档，而是给 **Agent** 执行任务用的 **SOP**

它通常描述：

- 任务开始前要确认什么上下文
- 应该按什么步骤执行
- 每一步应该优先使用什么工具
- 哪些文件、命令或行为不能碰
- 什么时候需要加载额外资料
- 最终输出应该是什么形式
- 必要时如何检查产物质量

例如文档处理 Skill 的 `SKILL.md` 正文可以写成：

```md
## 工作流程

1. 先确认文档目标、读者和格式要求
2. 生成或修改 docx 内容
3. 需要统一格式时，再读取 assets 中的模板
4. 文档生成后，调用 scripts/render_docx.py 渲染页面预览
5. 如果发现分页、字体、表格或图片问题，再读取 references/layout-rules.md
6. 根据预览结果修正文档，直到版式可用
```

这里的重点是：**指令层负责逻辑和按需加载策略**

`SKILL.md` 不需要把模板内容、排版规则和脚本源码全部写进去

它只需要告诉 **Agent**：

- 当前阶段该做什么
- 什么时候调用哪个脚本
- 什么时候读取哪个参考资料
- 什么时候使用哪个模板
- 什么时候检查结果是否可用

从编程类比看，`SKILL.md` 像模块入口文件或主函数

它定义控制流，但不会把所有依赖实现都内联到主函数里

#### 3.3、资源层：`scripts`、`references`、`assets`

资源层包含 **Skill** 执行任务时可能用到的材料

| 资源目录 | 作用 | 文档处理示例 |
| -------- | ---- | ------------ |
| `scripts` | 可执行脚本，处理稳定、重复、机械化任务 | `render_docx.py` |
| `references` | 参考资料，提供规范、规则和领域知识 | `layout-rules.md` |
| `assets` | 静态素材，提供模板、图片、样式或示例 | `report-template.docx` |

资源层的重点是==“**按需加载**”==，不是“默认全部加载”，也就是所谓的==“**渐进式披露**”==

例如：

- 进入“渲染检查”步骤时，再调用 `scripts/render_docx.py`
- 发现排版问题时，再读取 `references/layout-rules.md`
- 用户要求套用模板时，再使用 `assets/report-template.docx`

这就是指令层和资源层的关系：

- **指令层**：负责逻辑、顺序和按需加载策略
- **资源层**：负责执行材料、脚本能力和即时调用

这种结构能避免一个常见问题：把所有规则、资料、模板都塞进上下文，最后模型反而抓不住当前步骤真正需要的信息

#### 3.4、质量约束：写进 SOP 的完成标准

质量约束用于说明“做到什么程度才算可用”

它不一定是每个 **Skill** 的硬性组成，也不一定有单独文件

但对于工程任务来说，它非常重要，因为 **Agent** 容易完成表面产物，却漏掉必要检查

常见质量约束包括：

- 文件是否生成
- 格式是否符合要求
- 命令是否执行成功
- 截图或渲染结果是否正常
- 测试、构建、Lint 是否通过
- 输出是否覆盖用户要求的关键点

所以，更准确的说法不是“Skill 天然包含验证标准”，而是：

> **好的工程化 Skill 会把必要的完成标准写进 SOP，让 Agent 知道什么时候应该检查结果，而不是只凭主观判断宣布完成**

### 四、Skill 和其他概念的区别

#### 4.1、Skill 和 Prompt

最小 **Skill** 可以是一段 **Prompt**，但工程化 **Skill** 不应该只理解成“更长的 **Prompt**”

二者区别在于：

| 对比项 | 普通 Prompt | Skill |
| ------ | ----------- | ----- |
| 作用范围 | 当前一次请求 | 一类任务 |
| 复用方式 | 复制或改写提示词 | 按触发条件启用 |
| 内容结构 | 通常是一段连续指令 | `SKILL.md`、`scripts`、`references`、`assets` 组合 |
| 上下文策略 | 容易一次性塞入全部信息 | 按步骤精准读取资源 |
| 维护方式 | 分散在对话中 | 作为文件和目录结构维护 |

普通 **Prompt** 更像一次沟通，**Skill** 更像一个可维护的任务模块

它可以从一段 **Prompt** 开始，但逐步演化出：

- 触发规则
- `SKILL.md` **SOP**
- 可执行脚本
- 参考资料
- 模板素材
- 完成标准

#### 4.2、Skill 和 Tool

**Tool** 是外部能力接口，提供一个可以被调用的动作

**Skill** 是任务组织方式，规定如何组合这些动作完成目标

例如：

- 文件读取是 **Tool**
- 终端命令是 **Tool**
- 浏览器截图是 **Tool**
- “完成一次文档创建、渲染、检查和修正流程”是 Skill

**Tool** 更像函数，**Skill** 更像围绕目标组织起来的业务流程

#### 4.3、Skill 和 MCP

**MCP** 更偏协议层，解决 **Agent** 如何统一发现和调用外部工具、资源和提示词

**Skill** 更偏任务方法层，解决 **Agent** 如何完成一类具体任务

可以理解为：

- **MCP** 让外部能力以统一方式暴露出来
- **Skill** 告诉 Agent 在具体任务里如何使用这些能力

**MCP** 关注“能力接入”，**Skill** 关注“任务执行方法”

#### 4.4、Skill 和 Agent

**Agent** 是执行主体

**Skill** 是 Agent 可调用的能力说明、流程和资源组织方式

一个 **Agent** 可以使用多个 **Skill**

同一个 **Skill** 也可以被不同 **Agent** 使用，只要它们具备读取说明、调用工具和访问资源的能力

### 五、总结

生活化的理解：

> **Skill 就像 Agent 的专项菜谱，告诉它什么时候做这道菜、按什么步骤做、需要哪些工具和材料**

工程化的理解：

> **Skill 是给 Agent 使用的可复用任务模块，它用 `SKILL.md` 中的元信息决定何时触发，用 `SKILL.md` 正文定义 SOP，用 `scripts`、`references`、`assets` 提供按需调用的执行材料，并在需要时加入质量约束**
