## 用专家工作模式约束 Agent：从能力清单到可执行上下文

### 一、能力清单不能自动形成专家上下文

一个 **Agent** 工程发展到一定阶段后，通常会积累很多能力节点

- **Skills**：沉淀工作流、操作步骤、输出规范
- **Tools**：执行检索、采集、读写、调用外部系统等动作
- **Knowledge Base**：保存事实、历史材料、业务规则、客户记录
- **Data Surface**：承载原始证据、中间状态、草稿、审核产物
- **Quality Gate**：检查风险、证据、边界和交付质量

这些节点回答的是“系统有哪些局部能力”

其中 **skill** 通常描述某一种技能或工作流，例如：

- 如何做内容拆解
- 如何生成脚本
- 如何做质量复核

==但真实业务任务往往是**场景型目标**==，例如：

- 基于客户反馈规划内容
- 处理一个潜在线索”“把会议记录沉淀成后续跟进方案

这类目标需要的不只是某个单点技能，而是一个能够承接场景的专家上下文

它需要说明：

- 当前任务应该从哪个角色视角理解
- 属于哪个业务领域
- 应优先读取哪些知识
- 常用哪些技能组合
- 什么时候交接给其他流程
- 什么产物必须进入质量门

专家工作模式要解决的就是这个问题：==**把一类业务场景定义成可被 Agent 识别和进入的专家上下文，再由这个上下文组织相关 skill、tool、knowledge 和 quality gate**==

如果把所有路径都写成**硬编码状态机**，又会把 **LLM** 的语义判断能力压得太死，**用户意图稍微变化，流程就容易卡住或走错分支**

所以更实用的做法，是在「纯提示词」和「硬状态机」之间，加入一层工程级 **Soft Harness**

它不直接替代 **Agent** 推理，而是让 **Agent** 在推理前先进入正确的专家上下文，再决定如何组合 **skill**、**tool**、知识库和质量门

### 二、认知框架：Soft Harness 与 Hard Harness

**Harness** 可以理解为包裹 **Agent** 的工程约束系统

它不只是 **prompt**，也不只是工具调用代码，而是一组让 **Agent** 可控、可解释、可复用的约束

**Soft Harness** 主要通过「文本化」、「结构化」的工程规则约束 **Agent**

- 专家工作模式
- 触发意图
- 推荐工作流
- 工具路由规则
- 数据面边界
- 交接规则
- 质量门

它的目标是让 **Agent** 知道应该怎么做

**Hard Harness** 主要通过代码和运行时机制强制约束 **Agent**

- 状态机
- **Schema** 校验
- **Validator**
- 权限系统
- 审批流
- 自动化评测
- 运行时审计

它的目标是让 **Agent** 不能随便做，两者不是替代关系，而是递进关系

在需求还在变化、任务高度依赖语义判断、业务流程尚未完全稳定时，先用 **Soft Harness** 建立「专家上下文」和「可执行路径」，通常更轻、更快，也更符合 **LLM** 的工作方式

当某些路径变得高频、高风险、强合规或必须稳定复现时，再把其中的关键节点升级成 **Hard Harness**

### 三、设计范式：Expert Work Mode Pattern

**Expert Work Mode Pattern** 的核心不是**“分流”**，而是把一类业务场景定义成可进入、可执行、可交接、可复核的专家上下文

一个专家工作模式组成部分：

1. **工作场景**：它承接哪一类业务问题
2. **身份契约**：它用什么角色、租户、领域和知识范围理解任务
3. **触发意图**：什么样的用户目标会进入这个模式
4. **声明技能**：它默认组合哪些 **skill**
5. **工具边界**：它可以调用哪些 **tool**，以及这些工具只负责什么动作
6. **数据面**：原始证据、知识视图、草稿和审核产物分别放在哪里
7. **交接规则**：什么时候转给其他模式或流程
8. **质量门**：什么情况下必须复核后才能交付

一次具体任务穿过这个模式时，执行链路：

> 用户意图 -> 命中专家工作模式 -> 进入对应角色 / 领域 / 知识范围 -> 选择合适 **skill** -> 调用必要 **tool** -> 读写对应数据面 -> 必要时进入质量门 -> 输出交付物

#### 1. 工作场景：承接哪一类业务问题

一个 **Expert Work Mode** 描述的是一类工作场景

它不是用户这一次提出的具体任务，而是这类任务应该进入的专家上下文

例如：

- `content-strategist`：内容策略专家模式
- `customer-support`：客户支持专家模式
- `research-analyst`：研究分析专家模式
- `delivery-reviewer`：交付质量复核模式

用户说“基于客户反馈规划下周内容”，这是一个具体任务

这个任务可能命中 `content-strategist` 模式

#### 2. 身份契约：角色、租户、领域和知识范围

一个好的 **Expert Work Mode** 不应该只写“负责内容”

它需要有明确的身份契约

- `mode_id`：模式编号，用来稳定引用这个模式
- `role_id`：源角色标识，用来描述专家视角
- `tenant_id`：租户或业务主体，用来限定服务对象
- `domain`：领域，用来限定判断语境
- `knowledge_scope`：知识库路径或知识范围，用来限定证据来源

**这些 meta 信息不是装饰，而是专家上下文的边界**

它告诉 **Agent**：

- 我现在代表哪个业务主体
- 我应该用哪个领域的判断框架
- 我应该优先读取哪类知识
- 我不应该越过哪些上下文边界

#### 3. 触发意图：什么目标会进入这个模式

**Trigger Intent** 描述一类用户意图

它不是关键词匹配，而是语义触发条件

例如内容专家模式可以被这些意图触发：

- 用户要从客户反馈中提炼内容机会
- 用户要分析竞品内容或爆款内容
- 用户要生成选题池、内容 **brief** 或发布计划
- 用户要把研究结论转成对外内容草稿

当多个模式同时命中时，应该先进入编排或总控模式，拆解任务后再进入专业模式

#### 4. 声明技能：推荐能力组合，不是封闭白名单

**Declared Skills** 用来声明这个专家模式的**常见能力组合** <!--并不代表只能使用这些 skill-->

专家模式不是权限系统，它是 **Agent** 的工作语境

如果任务中发现资料不足，**Agent** 可以追加资料入库类 **skill**

如果任务涉及对外发布，**Agent** 可以追加质量门 **skill**

如果任务涉及系统故障，**Agent** 可以转向系统集成或排障类 **skill**

更准确的关系是：

**Expert Work Mode 提供专家上下文，Workflow Skill 提供具体操作流程**

#### 5. 工具边界：Tool 只承载动作能力

**Tool** 不应该承担业务判断，工具的职责应该收敛为明确动作：

- 查询客户反馈
- 检索知识库
- 读取文件
- 写入草稿
- 创建任务卡片
- 调用外部系统

业务判断应该由 **Agent** 在专家模式和 **skill** 的指导下完成

这样可以避免工具变成混乱的“业务大杂烩”，也方便后续测试、替换和权限控制

#### 6. 数据面：材料分别落在哪里

专家工作模式还需要知道数据面边界

典型数据面包括：

- `raw evidence`：原始证据，例如客户反馈、会议转写、日志、截图
- `knowledge view`：整理后的知识视图，例如用户画像、产品事实、方法论
- `draft`：待修改草稿，例如内容计划、客服回复、报告初稿
- `runtime state`：运行状态，例如任务卡片、处理日志、审核状态
- `review surface`：人工审核材料，例如质量门报告、风险标注

没有数据面边界时，**Agent** 很容易把原始证据、知识沉淀、对外草稿和运行日志混在一起

#### 7. 交接规则：什么时候转给其他模式或流程

专家工作模式不应该试图处理所有事情

它需要说明什么时候继续留在当前模式，什么时候把任务交给其他模式或流程

常见交接条件包括：

- 输入证据不足时，先转入资料采集或研究模式
- 用户目标过大时，先转入编排模式拆解任务
- 产物要对外发布时，转入质量门或审核模式
- 任务涉及系统故障时，转入系统集成或排障模式
- 当前模式无法覆盖任务目标时，回退到全局规划

交接规则的作用，是防止 **Agent** 在一个专家模式里硬做所有事情

#### 8. 质量门：处理高风险输出

不是所有输出都需要强复核

但这些场景必须有质量门：

- 对外发布内容
- 客户回复
- 价格、结果、交付承诺
- 代码变更完成声明
- 涉及法律、医疗、金融等高风险建议
- 使用客户案例、内部数据、敏感信息

质量门的职责不是重做全部工作，而是检查证据、风险、边界、语气、承诺和可交付性

### 四、完整示例：内容专家工作模式

下面用一个通用内容运营场景说明完整结构

先看一段接近工程约束文件的写法

```markdown
**内容策略专家**

模式编号：`content-strategist`
源角色标识：`strategist`
租户标识：`acme-brand`
领域：**B2B SaaS** 内容增长
知识库路径：`${BRAND_KNOWLEDGE_BASE}`
声明技能：`audience-research`、`content-briefing`、`campaign-planning`、`copy-review`

触发意图：

- 从客户反馈中提炼内容机会
- 分析竞品内容、行业内容或爆款内容
- 生成选题池、内容 brief、发布计划
- 把研究结论转成 **LinkedIn**、博客、邮件或社媒草稿
- 检查内容是否符合品牌定位和目标受众

优先加载：

- 先加载 `audience-research` 和 `content-briefing`，确认目标受众、内容机会、证据来源和内容 brief
- 需要形成发布计划时，加载 `campaign-planning`
- 需要对外发布或进入审核时，加载 `copy-review`

追加加载：

- 缺少客户反馈时，追加 `research-intake`
- 需要写正式发布稿时，追加 `copywriting`
- 需要生成任务排期时，追加 `content-calendar-planning`
- 需要复盘历史表现时，追加 `performance-analysis`

工具边界：

- `crm_query`：查询客户反馈、销售线索、常见问题
- `knowledge_search`：查询品牌定位、产品事实、目标用户、禁用表达
- `competitor_content_search`：查询竞品或行业内容样本
- `content_calendar_write`：写入内容计划
- `draft_store_write`：保存待审核草稿

数据面：

- `raw/feedback`：原始客户反馈、访谈摘录、销售记录
- `knowledge/brand`：品牌定位、产品事实、受众画像、禁用承诺
- `knowledge/content`：历史内容、表现复盘、内容方法论
- `drafts/content`：内容 brief、发布草稿、内容日历
- `reviews/content`：质量门检查、风险标注、修改建议

交接规则：

- 缺少事实证据时，交给 `research-intake`
- 用户要对外发布时，交给 `copy-review`
- 用户目标过大时，先交给 `orchestrator` 拆解任务
- 需要写入正式内容日历时，先确认周期、渠道、目标受众和审核人
- 涉及价格、客户案例、产品能力承诺时，必须标注证据来源

工作规则：

- 不编造客户案例
- 不承诺未经确认的产品能力
- 不把内部客户反馈原文直接对外发布
- 不绕过品牌语气、法务和隐私边界
- 不把单条客户意见当成整体市场结论
- 不在证据不足时给出确定性增长判断

输出要求：

- 输出内容目标、目标受众、依据来源、核心洞察、选题或内容计划
- 标注推荐渠道、草稿状态、待确认事项和风险提示
- 对外发布草稿必须经过 `copy-review`
```

具体任务路径：

```
用户输入：“基于最近客户反馈规划下周 5 条 LinkedIn 内容”
  -> 命中 content-strategist
  -> 读取 ${BRAND_KNOWLEDGE_BASE} 与客户反馈
  -> 使用 audience-research 提炼痛点
  -> 使用 campaign-planning 生成内容计划
  -> 使用 copy-review 检查事实依据和对外风险
  -> 输出内容日历、依据来源、待确认事项和风险提示
```

### 五、为什么这种设计值得沉淀

**Expert Work Mode Pattern** 的价值，在于把 **Agent** 的能力清单组织成可维护、可解释、可复用的专家上下文

- **把零散能力组织成专家上下文**
  - **Agent** 不再只是看到一堆 **skills** 和 **tools**，而是先判断“我现在应该以哪个专家模式工作”
- **保留 LLM 的语义判断能力**
  - 专家模式不是封闭白名单，**Agent** 仍然可以根据输入缺口追加其他 **skill** 或切换模式
- **让工程边界显性化**
  - 租户、领域、知识库、工具、数据面和质量门都被写进 **mode contract**，减少隐性假设
- **适合半结构化任务**
  - 内容运营、客服、销售、教育咨询、个人知识库、工程协作，都很难一开始写成稳定状态机，但很适合先写成专家工作模式
- **可以逐步升级为 Hard Harness**
  - 当某个专家模式里的路径稳定后，可以把其中的输入输出格式、审批节点、工具权限、质量检查升级为代码级约束

### 六、落地时应该注意什么

- **先定义专家场景，不要先堆工具**
  - 工具越多，越需要先回答“这些工具服务于哪些专家场景”
- **meta 信息要认真写**
  - `mode_id`、`role_id`、`tenant_id`、`domain`、`knowledge_scope`、`declared_skills` 这些字段决定了专家上下文是否清晰
- **skill 写流程，mode 写上下文**
  - 不要把所有步骤都塞进 **mode**
  - **mode** 负责定义专家身份、触发意图、交接规则和风险边界
  - **skill** 负责定义具体工作流、输入缺口、输出结构和检查项
- **tool 只做动作，不做业务判断**
  - 如果工具里混入太多业务判断，后续会很难测试、替换和复用
- **数据面必须显式区分**
  - 原始证据、知识视图、草稿、运行状态、审核产物应该分开承载
- **高风险输出必须有质量门**
  - 对外内容、客户回复、代码完成声明、价格承诺、结果承诺都不应该直接从主工作流输出
- **未命中专家模式时，要有 fallback 原则**
  - 没有命中明确模式，不代表 **Agent** 完全自由发挥
  - 它应该退回到全局边界、用户意图、可用 **skill** 描述、工具路由规则和证据要求来规划工作流

### 七、结论

**Expert Work Mode Pattern** 的核心价值，是让 **Agent** 在执行任务前先进入正确的专家上下文，而不是直接面对零散的 **skill** 和 **tool**

当系统开始拥有多个领域、多个知识库、多个输出场景和多个风险等级时，==**专家工作模式会成为关键组织层**==

它让工程文件从“说明 Agent 有哪些能力”，进一步回答：

**面对这类任务，Agent 应该以什么专家身份工作，读取哪些知识，调用哪些能力，把产物落在哪里，什么时候必须复核**
