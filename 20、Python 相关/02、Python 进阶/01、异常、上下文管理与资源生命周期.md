## 异常、上下文管理与资源生命周期

### 一、异常捕获

#### 1.1、语法与用法

**Python** 使用 `try / except / finally` 捕获和清理异常，整体接近 **JavaScript** 的 `try / catch / finally`

- `try`：放可能失败的代码
- `except SomeError as error`：捕获指定异常
  - `SomeError` 不是固定写法，而是某个具体异常类型的占位名
  - 常见具体异常：
    - `FileNotFoundError`：文件不存在
    - `ValueError`：值格式或取值不合法
    - `TypeError`：类型不符合预期
    - `KeyError`：读取 `dict` 中不存在的 key
    - `TimeoutError`：操作超时
- `except Exception as error`：捕获大多数普通异常
  - `Exception` 是大多数普通异常的共同父类，覆盖范围比具体异常更大
- `finally`：无论成功或失败都会执行
- `raise`：抛出异常

工程实践中，能预判具体错误时优先捕获具体异常；只有在任务边界需要统一兜底时，才使用 `except Exception as error`

#### 1.2、示例代码

```python
# Python
try:
    result = run_tool("read_file")
except FileNotFoundError as error:
    result = f"file missing: {error}"
finally:
    print("tool call finished")
```

```js
// JavaScript
try {
  const result = runTool("read_file");
} catch (error) {
  const result = `file missing: ${error.message}`;
} finally {
  console.log("tool call finished");
}
```

### 二、自定义异常

#### 2.1、语法与用法

自定义异常就是自己定义一个错误类型，通常继承 `Exception`

```python
class ToolError(Exception):
```

这里的 `ToolError` 是新定义的异常类，括号里的 `Exception` 是父类，表示 `ToolError` 属于普通异常体系

 <!--相当于 class ToolError extends Error {}-->

自定义异常的核心作用，是把底层错误转换成更贴近业务边界的错误

- 配置读取失败可以转换成 `ConfigError`
- 工具执行失败可以转换成 `ToolError`
- **provider** 请求失败可以转换成 `ProviderError`
- `raise NewError(...) from error` 可以保留原始异常链

#### 2.2、示例代码

```python
# Python
class ToolError(Exception):
    # 这里继承了 Exception，所以 ToolError 可以被 raise / except 处理
    pass #  Python 的空语句，意思是“这里什么都不做”


def read_tool_config(path: str) -> str:
    try:
        with open(path, "r", encoding="utf-8") as file:
            return file.read()
    except FileNotFoundError as error:
        # f 是 f-string，也就是格式化字符串，类似于 js 中的模板字符串
        raise ToolError(f"tool config not found: {path}") from error
```

```js
// JavaScript
class ToolError extends Error {}

function readToolConfig(path) {
  try {
    return readFileSync(path, "utf-8");
  } catch (error) {
    throw new ToolError(`tool config not found: ${path}`, { cause: error });
  }
}
```

### 三、with 与 context manager

#### 3.1、语法与用法

`with` 用于进入一个资源作用域，==离开作用域时自动执行清理逻辑==

它常用于管理“用完必须释放”的资源，例如文件、数据库连接、锁、临时目录

`with open(path) as file:` 可以拆成三部分理解

- `open(path)`：打开文件，返回一个文件对象
- `as file`：把这个文件对象绑定到变量 `file`
- `with` 代码块：在缩进范围内使用 `file`，代码块结束后自动关闭文件

#### 3.2、示例代码

```python
# Python
with open("session.json", "r", encoding="utf-8") as file:
    # file 是 open(...) 返回的文件对象
    content = file.read()
```

上面代码大致等价于手写 `try / finally`

```python
# Python
file = open("session.json", "r", encoding="utf-8")

try:
    content = file.read()
finally:
    file.close()
```

```js
// JavaScript
const content = readFileSync("session.json", "utf-8");
```

### 四、资源生命周期

#### 4.1、语法与用法

**资源**可以理解为“向操作系统或外部服务借来的东西”：

- **文件句柄**：向操作系统借文件读写能力，用完要关闭
- **数据库连接**：向数据库借连接，用完要关闭或归还连接池
- **子进程**：启动另一个进程执行命令，用完要等待、终止或回收

资源生命周期通常包含四个阶段：

- 创建资源：例如 `open(...)`、连接数据库、启动子进程
- 使用资源：读取文件、执行 SQL、读取子进程输出
- 处理失败：超时、权限错误、命令失败、连接断开
- 释放资源：关闭文件、归还连接、等待或终止子进程

如果只创建资源、不清理资源，会出现文件句柄泄漏、连接耗尽、子进程残留、请求永久卡住等问题

#### 4.2、示例代码

```python
# Python
import subprocess


def run_command(command: list[str], timeout: int = 30) -> str:
    # 启动子进程，并等待它结束
    completed = subprocess.run(
        command,
        # 收集子进程的标准输出和标准错误（stdout、stderr），避免输出直接打到当前终端
        capture_output=True,
        # 把输出从 bytes 解码成 str
        text=True,
        # 最多等待 timeout 秒，避免命令永久卡住
        timeout=timeout,
        # 退出码非 0 时抛出 CalledProcessError
        check=True,
    )
    # 子进程已经结束，输出被收集到 `completed` 里
    return completed.stdout
```

这段代码的生命周期是

1. `subprocess.run(...)` 启动子进程
2. 当前 **Python** 代码等待子进程结束
3. 子进程正常结束时，返回 `CompletedProcess`
4. 子进程失败、超时或找不到命令时，抛出异常
5. 函数返回时，调用方只拿到整理后的 `stdout`

```js
// JavaScript
import { execFile } from "node:child_process";
import { promisify } from "node:util";

const execFileAsync = promisify(execFile);

async function runCommand(command, args, timeout = 30_000) {
  const result = await execFileAsync(command, args, { timeout });
  return result.stdout;
}
```

这里的 **JavaScript** 示例用了 `async / await`，是因为 **Node.js** 常把子进程结果包装成 `Promise`

它和 **Python** 示例关注的是同一个生命周期问题：启动命令、等待结果、限制超时、收集输出、把失败变成可处理的错误
